# AI_CONTEXT.md

# I Wrote This Instead

## AI Assistant Context Document

---

# 1. Project Identity

Project Name:

```
I Wrote This Instead
```

Project Type:

```
Minimalist poetry archive
```

Primary Purpose:

```
Provide a quiet, personal space for sharing writing.
```

---

# 2. What This Project Is

This project is a digital notebook.

It exists to preserve writing.

The website is not a blog platform.

It is not a social network.

It is not a content machine.

It is a collection of poems.

---

# 3. Core Feeling

Every implementation decision should preserve this feeling:

Someone left their notebook open, and trusted you enough to read it.

---

# 4. Emotional Direction

The website should feel:

- personal
- honest
- vulnerable
- reflective
- quiet
- timeless

The website should not feel:

- corporate
- optimized
- commercial
- trendy
- artificial

---

# 5. Design Summary

Visual identity:

```
Warm paper
+
Black text
+
Monospace writing
+
Large whitespace
+
Minimal interface
```

The website should resemble:

- a printed poetry book
- a personal journal
- a typed manuscript

---

# 6. Technical Identity

Framework:

```
Next.js
```

Language:

```
TypeScript
```

Styling:

```
CSS Modules
```

Content:

```
Markdown files
```

Deployment:

```
Vercel
```

---

# 7. Architecture Rules

Remember:

Markdown is the source of truth.

The filesystem is the CMS.

Do not introduce:

- databases
- APIs
- authentication
- unnecessary state
- unnecessary dependencies

---

# 8. Rendering Rules

Prefer:

Server Components.

Static generation.

Build-time rendering.

Avoid:

Client-side rendering.

Runtime fetching.

---

# 9. Content Rules

Never modify poem content.

Never summarize poems.

Never rewrite poems.

Never automatically alter formatting.

The author's writing must appear exactly as provided.

---

# 10. Design Rules

Always maintain:

Background:

```
#F7F2E8
```

Text:

```
#111111
```

Content width:

```
700px
```

Typography:

```
Monospace for poems
Inter/system fonts for UI
```

---

# 11. Feature Evaluation

Before adding a feature ask:

Does this help someone read?

Does this help someone discover writing?

Does this preserve the feeling of the project?

If not:

Do not add it.

---

# 12. Features To Avoid

Do not introduce:

- likes
- comments
- followers
- advertisements
- engagement metrics
- social feeds
- unnecessary analytics

---

# 13. Coding Style

Prefer:

Simple code.

Readable code.

Small components.

Explicit types.

Clear naming.

---

Avoid:

Over-engineering.

Large abstractions.

Complex state management.

---

# 14. Component Philosophy

Components should be:

Small.

Focused.

Reusable.

Examples:

```
Navbar

Footer

PoemCard

BackButton

PreviousNext
```

---

# 15. Future Features

Allowed future ideas:

- dark mode
- tags
- search
- collections
- audio readings
- timeline
- newsletter

However:

Future features must preserve simplicity.

---

# 16. When Asked To Modify The Project

Before writing code:

Understand:

1. Why the change exists.

2. How it affects the reading experience.

3. Whether it matches the project's identity.

---

# 17. When Generating Code

Always:

Use TypeScript.

Use existing patterns.

Follow current folder structure.

Respect existing styling.

Avoid introducing new dependencies without justification.

---

# 18. When Debugging

Do not immediately rewrite.

First:

1. Identify the problem.

2. Explain the cause.

3. Apply the smallest fix.

---

# 19. When Adding Pages

New pages should:

- have a clear purpose
- fit existing navigation
- use existing styles
- maintain simplicity

---

# 20. When Adding Components

Ask:

Can an existing component handle this?

Avoid duplicate UI patterns.

---

# 21. Quality Checklist

Before completing work:

Check:

✓ Does it fit the design?

✓ Does it preserve simplicity?

✓ Is it mobile friendly?

✓ Is it accessible?

✓ Does it avoid unnecessary complexity?

✓ Would this still make sense years from now?

---

# 22. The Most Important Rule

The website is not the product.

The writing is the product.

The code exists to protect the writing.

When uncertain:

Choose the quieter option.

Choose the simpler option.

Choose the option that lets the words speak.

---

# End Context

This project is a place for words that needed somewhere to exist.

Build accordingly.
